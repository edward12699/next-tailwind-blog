## what‘s the problem？

Have you ever been reading an article online when something suddenly changes on the page

## how it happens

Unexpected movement of page content usually happens because resources are loaded asynchronously or DOM elements get dynamically added to the page above existing content. 
The culprit might be an image or video with unknown dimensions, a font that renders larger or smaller than its fallback, or a third-party ad or widget that dynamically resizes itself.

What makes this issue even more problematic is that how a site functions in development is often quite different from how users experience it. 
Personalized or third-party content often doesn't behave the same in development as it does in production, test images are often already in the developer's browser cache, and API calls that run locally are often so fast that the delay isn't noticeable.

## what is CLS

CLS is a measure of the largest burst of layout shift scores for every unexpected layout shift that occurs during the entire lifespan of a page.
Layout shifts are defined by the Layout Instability API, which reports layout-shift entries any time an element that is visible within the viewport changes its start position (for example, its top and left position in the default writing mode) between two frames. Such elements are considered unstable elements.
Note that layout shifts only occur when **existing elements** change their start position. If a new element is added to the DOM or an existing element changes size, it doesn't count as a layout shift—as long as the change doesn't cause other visible elements to change their start position.

## transform property allows you to animate elements without triggering layout shifts:


Instead of changing the height and width properties, use transform: scale().
To move elements around, avoid changing the top, right, bottom, or left properties and use transform: translate() instead.

transform: scale() 和translate 不管你缩放或者移动多少，在界面上占据的始终只是一倍的空间，不影响其他元素

建议使用web-vitals，因为它帮我们处理了很多复杂的场景，比如
   1. 页面在后台加载，则不应该报告cls，因为对用户没有产生影响 (可利用visibilitychange event来感知)
   2. 如果页面从cache中得到，cls 应该重制为0
   3. iframe的情况 PerformanceObserver不会计算iframe的情况，但是iframe应该被考虑进来，因为它同样是页面以及用户体验的一部分
   4. 一些额外的情况增加了cls的复杂度，比如一个页面从不关闭，则这个页面的生命周期很难界定,  或者是在某些手机端的操作系统中，浏览器通常不会为后台选项卡运行页面卸载回调，因此很难报告“最终”值

3. LCP代表"大量内容的绘制"（Largest Contentful Paint），是一项用于衡量网页加载性能的指标。LCP衡量的是从页面开始加载到最大内容元素完全绘制在用户视图中所需的时间。
    最大内容元素通常是指页面中最大、最显著的可见内容，例如图片、视频、文本块或其他DOM元素。LCP的目标是确保这些重要内容能够尽快绘制出来，以提供更好的用户体验和视觉反馈。
    较短的LCP时间意味着页面的关键内容能够更快地展示给用户，减少了用户等待的时间。通常，一个优秀的LCP时间应该在1.5秒内。
    为了改善LCP，可以采取一些优化措施，例如：
      1. 增加服务器响应速度，确保内容能够快速加载。
      2. 优化图像大小和格式，减少图像文件的下载时间。
      3. 延迟加载非关键资源，优先加载与LCP相关的重要内容。
      4. 最小化渲染阻塞，例如通过异步加载脚本或样式表来避免阻塞主要内容的绘制。
4. FID代表"首次输入延迟"（First Input Delay），是衡量网页交互性能的指标之一。它用于评估用户首次与页面进行交互时所经历的延迟。
    FID测量的是从用户首次与页面交互（例如点击按钮、选择下拉菜单）到浏览器实际响应该交互的时间间隔。较短的FID意味着用户能够更快地与页面进行互动，并获得即时的反馈。
    FID的目标是保持在100毫秒以下，因为在此范围内，用户会感觉到交互是即时的。如果FID超过300毫秒，用户可能会感到延迟，这会影响用户体验。
    要改善FID，可以采取以下优化措施：
    1. 减少主线程阻塞：避免JavaScript、样式表或其他资源的长时间执行，以防止阻塞用户交互。
    2. 延迟加载非关键脚本：将非关键脚本推迟加载，确保主要交互元素能够优先加载。
    3. 使用Web Worker：将计算密集型任务转移到Web Worker线程，以减少主线程阻塞。
    4. 压缩和优化JavaScript：减小JavaScript文件大小，优化其性能，提高执行效率。
